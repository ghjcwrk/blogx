## はじめに

---

多くのプロジェクトでは `Dev Containers` を単に「開発環境の起動手段」として扱います

`Dockerfile` があって、`Deno` や `Bun` が入り、あとは起動して開発を始める──

しかし私は `Dev Containers` をもっと広く捉えています

> それは「開発体験全体を IaC（Infrastructure as Code）で管理するための第一歩」

なぜなら**私たちの開発体験をもっとも壊しやすくするのはローカル環境の差異**だからです

- 誰かの `zshrc` にだけ `alias` があり

- `bun i` の順番で挙動が変わり

- `task` のバージョンが違って実行に失敗

これらは一見すると些細な差異でしょう。しかし開発初期の「つまずき」やチーム内の不整合の多くはこうした環境差から生まれています

ならばそれらを**コードで構成し、コンテナとして固定する**

それがすべてのプロジェクトにおける `Dev Containers` 運用の出発点であるべきと考えます

---

## すべての差異を吸収しようとしない勇気

---

Docker は万能か？ と訊かれたら私はこう答えます

いいえ、万能ではありません

ホスト OS の違い、たとえば macOS / Windows / Linux、すべてを吸収できるとは限りませんし、とくに GUI 操作やファイルパスまわりでは差異が生じやすくなります

しかし私は **「自分の使っている OS（さらにいえば CPU アーキテクチャ）で動くことを保証する」** というレベルであれば、**それは十分に現実的で、低コストで実現可能なものと捉えています**

Web 開発の現場では**半数以上が macOS ユーザー**です

私は自分が macOS を使っている以上、**同環境の相手に対してはある程度の動作保証が可能です**

では残りの Windows / Linux ユーザーへのアプローチは？

> 本当にチームメンバーに Windows を使っている人がいるなら
> 
> - その人の協力を得て、**デュアル保証の体制を築く**
> 
> - 本当に必要であれば AWS にでも ShadowPC にでも一時インスタンスを立てて検証する
> 
> \* ローカル仮想環境ではアーキテクチャは同一なため、検証環境としては最適ではありません

それでいいのだと思います

環境の差異を「最初からすべて吸収しよう」と構えるのではなく（そしておそらくそれは YAGNI）

まずは **IaC による土台を築く**こと。そして必要があればそこに**人の知恵や協力を重ねていく**こと

そこさえ出来あがっていれば、ツール選定から何から何まで**すべてがチームの議論の対象になる**。これこそが IaC 化の成果でもあると考えます

## 環境が語れるものとなったとき

---

たとえば

- bun -F はアップデートで壊れやすい。実績十分な pnpm を代わりに導入しないか？

- HTTPie よりも xh の方がシンプルで認知負荷が低い、我々のケースでは xh が最適ではないか？

- ある問題を解決してくれるかもしれないツールを発見した、皆で試してみないか？

そうした議論や技術的コミュニケーションが自然と生まれるようになるでしょう

今まではプロダクト構成やコードに閉じて、それのみをレビュー対象にしていた私たちが、直接的か間接的かを問わず、それらを形作るものすべてをその議論対象に据え、やがて収束させ、本当の意味で「知の共有」を行えるよう自然と変化します

情報の感度、関心、自ら選択したり与えられたロールは人によって異なります

- ユーザーに共感され、保守も発展も可能なコンポーネントをつくった人も

- そこに肉付けを行うための DB 基盤、API ルートを設計・実装した人も

- 彼らが安心・迅速に開発に取り組める基盤を整備した人も

同様に評価されるべき対象です

そしてフロントエンドもバックエンドも DevOpts もあえて垣根を設けるメリットが薄くなっていると、そう感じるならば、やはりまずは Dev Containers から始めるべきだと私はそう強く思います

`Dev Containers` はそんな「開発環境の民主化」の第一歩になりうると、私は信じています

## .devcontainer/ の構成を見てみる

---

まずは Dev Containers のすべての基点となる `.devcontainer/` ディレクトリの構成を紹介します

ここに全開発体験を閉じ込め、かつ共有可能な形式にすることで、プロジェクトを誰でもすぐ始められる状態にします

```zsh
/workspace
├── .devcontainer
|  ├── deno.json # Deno 関連の設定ファイル
|  ├── deno.lock # 使用ツールの再現性を担保する
|  ├── devcontainer.json # VSCode Dev Containers の起点
|  ├── docker-compose.yml # 各サービスを統合し、起動順や依存関係を定義する
|  ├── lib # 各サービスの entrypoint.ts から使用される共通ロジック
|  |  └── cmd.ts
|  └── services # サービスとして切り出したいものはここに入れる、基本的には SOLID 原則
|     ├── setup
|     |  ├── Dockerfile # OS・ランタイム定義（ベース環境）
|     |  └── entrypoint.ts # 起動時に一度だけ走る初期化スクリプト
|     └── shell
|        ├── .zshrc # shell の挙動を統一する（aliasなど）
|        ├── Dockerfile
|        ├── entrypoint.ts
|        └── starship.toml
├── .env
├── .gitignore
├── .vscode
|  ├── extensions.json
|  └── settings.json
```

この構成は決して決まりきったものではありません。プロジェクトやチームに応じて最適な形は自然と変わっていきます。ここで示しているのは説明のために簡素化した例にすぎません

実際にはプロダクトの性質やスケールに応じ、リバースプロキシ（HAProxy, Nginx）やキャッシュ層（Dragonfly, Redis）、データベース、さらには Rust / Go / Python / FFmpeg / Playwrite / LLM 基盤といったバックエンドサービス群がコンテナとして組み込まれることもあります

それでも私が裁量の及ぶ範囲でこの構成に近い形を提案・採用しているのは、以下のような明確な思想に基づいてのことです：

- **VSCode を前提とせず、単体で本番運用にも耐える Docker 環境を設計の主軸とする**

- **開発開始から即座に `task start` や `bun dev` が動作する状態を保証する**

- **開発に使う全ツールをコードとして定義し、透明性と再現性を担保する**

つまりこの構成は「すぐに動いて、すぐに共有できる」開発体験を支える、もっとも小さくて確実な単位と言えるでしょう

## 実践：entrypoint.sh を TypeScript で書く理由

---

従来の Dev Containers では、初期化処理を entrypoint.sh で行うのが一般的でした

しかし私たちは Deno + TypeScript でこれを置き換えています

なぜシェルスクリプトでは不十分なのか

多くのプロジェクトでは、開発環境の初期化をこう書きます：

```zsh
// サンプルコードを書く
```

一見すると問題ないように見えるでしょう

しかしこのような処理を entrypoint.sh で書くと：

- エラーハンドリングが困難　bash の成功・失敗判定は複雑で、どこで何が失敗したのかを特定しにくい

- ログ出力が不統一　echo と printf が混在し、実行状況の把握が困難

- 条件分岐が読みにくい　環境変数の存在チェックや文字列比較の構文が直感的でない

- 型チェックが存在しない　typo やオプションミスに気づけない

これらは開発環境構築でもっとも避けたい「何が起きているのか分からない」状況を作り出します

---

### TypeScript 化すると何が変わるか

---

同じ処理を TypeScript で書き直すとこうなります：

```ts
// サンプルコードを書く
```

この違いは明確です：

- エラーが明確　exec() の失敗は Promise rejection として扱え、どの段階で何が失敗したかが即座に分かる

- 型安全　typo やオプションミスを IDE・LSP が検出し、実行前に問題を発見できる

- テスタブル　関数として切り出せるため、単体テストや部分実行が可能

- 保守性　複雑な条件分岐も読みやすく記述でき、将来の拡張や変更に対応しやすい

> なぜ Deno なのか
> 
> Node.js でも TypeScript は書けますが、Deno を選ぶ理由があります：
> 
> ゼロコンフィグ　tsconfig.json や package.json が不要で、TypeScript がそのまま動く
> 
> 標準ライブラリが充実　ファイル操作、HTTP リクエスト、プロセス実行が標準で提供される
> 
> セキュリティ　明示的な権限管理（--allow-*）により、意図しない操作を防げる
> 
> シングルバイナリ　依存関係の管理が簡潔で、コンテナ環境との相性が良い
> 
> もっとも重要なのは、開発環境の初期化処理に Node.js を持ち込むと「Node.js の環境構築が必要」という本末転倒が起きることです
> 
> Deno なら「Deno 自体のインストール → 即座に TypeScript が動く」という最短経路を辿れます

この共通化により：

- ログ形式の統一　絵文字付きで視認性が向上し、実行状況が直感的に把握できる

- エラーハンドリングの標準化　すべてのコマンド実行で一貫したエラー処理が適用される

- 実行結果の透明性　何が起きているかが常に明確で、デバッグが容易

従来の entrypoint.sh は「動けばいい」という発想でした

しかし TypeScript 化することで、開発環境の初期化処理は「設計されたコード」になります

それは保守可能で、テスト可能で、拡張可能なものです

新しいツールを追加したい時、環境変数をチェックしたい時、条件分岐を追加したい時

すべてが型安全な関数として、IDE の補完とともに書けるようになります

これは単なる技術選択ではありません

開発環境すらコードとして扱うという思想の実践です

.devcontainer/ というディレクトリには、単なるツールの設定ファイル以上のものが詰まっています

それはすなわち、

- 開発者が「何を共通化し、何を個別に委ねるか」を定義した思想の写しであり、

- 新たに参加するメンバーが最初に触れる「開発体験の最前線」であり、

- チーム全体が「議論できる環境」を持つための出発点でもある

ということです

この小さな構成体に、プロジェクトの理念や将来の拡張可能性までもが反映されている

そう思えるなら、それはもう「設定」ではなく「設計」であり、「環境」ではなく「土台」です
